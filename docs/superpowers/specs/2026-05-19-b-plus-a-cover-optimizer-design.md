# 2026-05-19 B+A 覆盖优化实验模式设计

## 目标

- 新增一个并行实验模式，在不替换现有 `team` 主链路的前提下验证 `B+A` 方向。
- 将系统目标从“提高单点预测命中”切换为“在固定 5 注预算下优化综合覆盖表现”。
- 建立严格但务实的验证护栏：实验模式必须相对条件随机基准获得稳定 uplift，才继续推进。

## 背景结论

- 当前系统的主要问题不是缺少更多调参项，而是上游信号源同质化、核心池奖励共识、下游固定矩阵进一步放大信息损失。
- 蓝球虽然已经拆成独立引擎，但仍主要服务于线性融合后的 TopN 候选，不适合作为单点命中工具，更适合作为覆盖候选池来源。
- 现有多轮迭代主要修补链路与口径一致性，没有从“整体 5 注组合优化”角度重构出票器。

## 设计范围

- 修改 `predict.py`、`project_config.py`、`analyze_archive.py`、`README.md`、`SKILL.md`、`AGENTS.md`。
- 更新 `test_predict.py`、`test_analyze_archive.py`。
- 保留现有 `team`、`team-backtest`、补丁回灌、归档分析主链路。
- 不恢复 `LSTM/TensorFlow`，不扩增专家集合，不把实验模式直接设置为默认模式。

## 非目标

- 不承诺或暗示彩票存在稳定可预测性。
- 不在第一阶段重做现有专家体系。
- 不在第一阶段要求统计显著性检验作为硬门槛。
- 不把实验模式结果写入现有 `prediction_archive` 默认归档。

## 总体架构

实验模式新增一条并行链路：

`专家提案 -> 候选分布整理层 -> 覆盖优化出票层 -> 实验回测/基准验证层`

与现有主链路对照：

`专家提案 -> 核心池 -> 固定矩阵 -> 归档/回测`

### 候选分布整理层

- 复用现有专家提案结果与蓝球引擎结果。
- 将原先偏向“共识聚合”的池构建逻辑转成“候选评分 + 结构标签”的形式。
- 红球输出不仅包含分数，还包含以下标签：
  - 来源专家集合
  - 专家贡献分布
  - 所属区间
  - 奇偶属性
  - 和值贡献区间
  - 冷热/遗漏类别
- 蓝球输出不仅包含分数，还包含以下标签：
  - 高分主桶
  - 冷号探索桶
  - 反转/回归桶

### 覆盖优化出票层

- 不使用固定矩阵行作为实验模式出票核心。
- 将 5 注视作一个整体优化问题，逐注构造，目标是最大化组合整体收益而非单注局部得分。
- 红球采用启发式搜索生成 5 注：
  - 第 1 注偏向高综合得分且结构合法的组合
  - 第 2-5 注偏向补足前面未覆盖到的号码、对子、区间模式和和值区间
- 蓝球采用分桶覆盖与注间去重策略，而非统一追 Top1。

### 实验回测/基准验证层

- 同一批历史样本同时跑三条链路：
  - 现有 `team`
  - 新增实验模式
  - 条件随机基准
- 输出统一的综合指标与相对 uplift，用于判断实验模式是否值得保留。

## 核心目标函数

实验模式以综合指标为主目标，四个子指标共同决定表现：

- `best-of-5 红2+率`
- `best-of-5 红3+率`
- `蓝球池命中率`
- `组合分散度`

### 综合分建议

实验模式内部使用可解释的加权综合分，不直接等于开奖真实收益：

- 红 `2+` 覆盖为主指标，权重最高
- 红 `3+` 作为更高质量命中的补充指标
- 蓝球池命中率用于评估候选覆盖有效性
- 组合分散度用于惩罚 5 注之间的过度重合

默认建议权重：

- 红 `2+`：`0.40`
- 红 `3+`：`0.25`
- 蓝球池命中率：`0.20`
- 组合分散度：`0.15`

这些权重进入实验模式配置，可在后续回测中调整，但第一阶段固定默认值。

## 红球候选评分

### 现状问题

- 当前核心池会对跨 Agent 共识球追加奖励，这会放大同质化。
- 当前固定矩阵只能对已有候选做裁剪和重排，无法主动提升边际覆盖。

### 目标改法

- 保留专家贡献分数，但降低“共识越高越好”的倾向。
- 将共识奖励替换为“边际覆盖价值”：
  - 如果某号码虽然共识高，但只会加强已有热点堆积，则不额外加分
  - 如果某号码能带来新的结构覆盖或降低组合相似度，则获得额外加分

### 红球单号评分组成

- 专家基础贡献分
- 独立来源奖励
- 结构补位奖励
- 过度共识惩罚

其中：

- 独立来源奖励：鼓励来自少数但相对独立的信号源
- 结构补位奖励：鼓励能补足区间、奇偶、和值、冷热分布缺口的号码
- 过度共识惩罚：对已经高度拥挤的热点号适度降权

## 蓝球覆盖策略

### 设计原则

- 蓝球在实验模式中不追求“单个最可能号码”。
- 蓝球引擎主要作为候选打分器与分桶器，而不是单点决定器。

### 分桶方案

- 主桶：蓝球引擎高分号码
- 探索桶：高遗漏、冷号、近期未覆盖号码
- 反转桶：奇偶回归、区间回归、均值回归倾向号码

### 分配规则

- 5 注蓝球优先覆盖不同桶，不允许全部集中在同一桶。
- 优先避免重复蓝球，除非候选池实在过窄。
- 若主桶过强，可允许 1 个重复高置信蓝球，但默认仍以覆盖优先。

## 覆盖优化出票器

### 输入

- 红球候选池与分数
- 红球结构标签
- 蓝球候选池、蓝球分数与桶信息
- 目标注数，固定为 `5`

### 输出

- 固定 `5` 注 `6+1`
- 每注附带解释信息，说明该注主要在补哪些覆盖维度

### 红球搜索策略

- 使用启发式逐注生成，不引入过重的全局优化框架。
- 第一阶段采用“候选生成 + 贪心挑选 + 重叠惩罚”方式，避免实现复杂度过高。

#### 候选生成

- 从红球池中生成一批候选红球组合。
- 候选生成兼顾：
  - 单号综合分
  - 基础结构合法性
  - 号码去重覆盖潜力

#### 贪心挑选

- 第 1 注选择单注综合分最高的组合。
- 第 2-5 注每次选择“加入现有票集后总收益最高”的组合。

#### 注间惩罚

- 对与已有票重合 4 个及以上红球的候选组合显著降分。
- 对重复对子、重复三连结构、重复区间模板进行额外惩罚。
- 对能补齐冷门区间或扩展和值覆盖范围的组合给予奖励。

## 条件随机基准

### 设计原则

- 基准不能是无条件乱选，否则对照不公平。
- 基准必须与实验模式处于相同预算和相似约束条件下。

### 基准定义

- 使用相同红球池规模、相同蓝球池规模、相同注数。
- 使用相同的基础结构约束。
- 在这些约束下做随机出票。

### 比较口径

- 对比实验模式与条件随机基准的综合指标 uplift。
- 对比现有 `team` 与条件随机基准的 uplift。
- 最终判断不是“实验模式绝对多高”，而是“实验模式是否稳定优于条件随机”。

## 实验模式 CLI

第一阶段新增并行入口，建议采用以下能力之一：

- 新增 `--mode team-cover`
- 或新增 `--experimental-cover`

为减少现有 CLI 冲突，推荐复用 `--mode` 并引入 `team-cover`。

建议同步提供：

- `python predict.py --mode team-cover --num 5`
- `python predict.py --team-cover-backtest --backtest-cycles 36 --seed 42`

说明：

- `team-cover` 只做实验输出，不默认写入现有 `prediction_archive`
- `team-cover-backtest` 默认输出对照报告，不写归档

## 回测输出设计

新增实验回测报告，至少包含以下三组结果：

- `team`
- `team_cover`
- `conditional_random`

每组结果至少输出：

- 样本数
- 平均综合分
- 红 `2+` 命中率
- 红 `3+` 命中率
- 蓝球池命中率
- 最终蓝球命中率
- 组合平均重叠度

对比字段至少包含：

- `team_cover_vs_random_uplift`
- `team_vs_random_uplift`
- 子指标 uplift 明细

## 与现有链路的关系

### 保持不变

- `team` 仍是默认主模式
- 现有固定矩阵出票逻辑继续存在
- 现有归档分析、补丁导出、学习链路继续服务主模式

### 需要清理的语义问题

- `preferred_rows` 默认全量写入会短路 `row_weights` 动态排序分支，这会造成“矩阵学习存在但默认不生效”的歧义。
- 第一阶段应修正文档与默认配置，使现有 `team` 链路语义自洽。
- 该修复不作为实验模式成功的前提，但应纳入同一轮重构，以避免后续分析误判。

## 测试策略

### 单元测试

- 实验模式固定输出 `5` 注
- 蓝球分桶分配与默认去重行为正确
- 红球注间高重叠惩罚生效
- 条件随机基准输出结构与实验模式一致

### 回测测试

- 实验回测报告正确包含三组结果
- 对比 uplift 字段存在且格式稳定
- 样本不足时不输出误导性结论

### 回归测试

- 现有 `team` 主模式行为不被实验模式破坏
- 现有 `--team-backtest` 口径保持兼容

## 风险与约束

- 覆盖优化只能提升“组合效率”，不意味着存在真实预测能力。
- 若候选池本身没有有效信息，实验模式最多做到优于条件随机，而非长期显著预测未来。
- 实验模式若约束过严，可能导致候选搜索退化或输出过于保守，因此约束必须采用软惩罚而不是硬阻断。

## 实施顺序

### 第一阶段：实验模式骨架

- 新增实验模式 CLI 与配置入口
- 新增实验模式回测入口
- 保持现有主模式完全不变

### 第二阶段：红球覆盖优化出票器

- 新增候选分布整理
- 实现候选生成、贪心挑选和重叠惩罚
- 完成每注解释信息输出

### 第三阶段：蓝球分桶覆盖

- 为蓝球引擎结果增加桶语义
- 实现 5 注蓝球覆盖分配逻辑

### 第四阶段：条件随机基准与 uplift 报告

- 实现条件随机基准出票器
- 新增并排对比回测报告

### 第五阶段：文档与现有链路语义清理

- 更新 `README.md`、`SKILL.md`、`AGENTS.md`
- 修复 `preferred_rows` / `row_weights` 默认语义歧义

## 验收标准

- 实验模式在统一 walk-forward 回测中，综合指标相对条件随机基准表现出稳定正 uplift。
- 实验模式不会破坏现有 `team`、`team-backtest` 和归档分析链路。
- 新增输出具备解释性，能说明每注在补哪些覆盖维度。

## 暂不执行事项

- 不做专家体系重写
- 不做统计显著性强校验
- 不把实验模式结果并入主归档
- 不在第一阶段引入复杂元启发式算法或大规模搜索框架
